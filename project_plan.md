# План работ и организационная структура проекта

## 1. Этапы / области работ
1. Discovery
2. Design
3. Development
4. QA
5. Release

## 2. Оценки длительности по этапам и ключевым задачам
> Формат: этап / задача — оценка длительности — ответственный владелец

### Discovery (1–2 недели)
- Анализ требований и целей — 3–5 дней — Product Manager
- Сбор и приоритизация стейкхолдеров — 2–3 дня — Product Manager
- Формирование product backlog и roadmap — 2–4 дня — Product Manager

### Design (2–3 недели)
- Архитектурное проектирование — 5–7 дней — Solution Architect
- UX/UI прототипирование — 5–8 дней — UX/UI Designer
- Проработка нефункциональных требований — 3–5 дней — Tech Lead

### Development (4–8 недель)
- Настройка инфраструктуры/репозитория — 2–4 дня — DevOps Engineer
  - Блокирует старт backend- и frontend-разработки (CI/CD, окружения, база, репозиторий).
- Backend API и интеграции — 10–15 дней — Backend Engineer
  - Требует готовой инфраструктуры/репозитория.
  - Обеспечивает стабильные API и интеграции для UI и QA.
- Frontend и UX-сценарии — 12–18 дней — Frontend Engineer / UX
  - Требует доступных и стабильных backend API.
- Реализация ключевых фич (MVP) — 15–25 дней — Engineering Team
  - Выполняется после базовой инфраструктуры и доступности API.

### QA (2–3 недели)
- Подготовка тест-плана и сценариев — 3–5 дней — QA Lead
- Функциональное тестирование — 7–10 дней — QA Team
- Регрессионное тестирование — 3–5 дней — QA Team

### Release (1–2 недели)
- Подготовка релиза (релиз-ноты, чеклисты) — 2–3 дня — Release Manager
- Деплой на прод — 1–2 дня — DevOps Engineer
- Пострелизный мониторинг — 3–5 дней — SRE/DevOps

## 3. Реестр рисков
| Риск | Вероятность | Влияние | Меры реагирования |
| --- | --- | --- | --- |
| Неопределенность требований на старте | Средняя | Высокое | Формализовать требования, проводить регулярные демо и согласования |
| Задержки интеграций со сторонними системами | Средняя | Среднее | Раннее согласование API, буфер по срокам |
| Недостаток ресурсов/людей | Средняя | Высокое | Резервный план, привлечение внешних подрядчиков |
| Регрессии при частых релизах | Высокая | Среднее | Автоматизация тестов, регрессионный набор |
| Низкая производительность/масштабируемость | Низкая | Высокое | Нагрузочное тестирование и профилирование |

## 4. Состав команды / ролей (для назначения R/A по RACI)
- Product Manager
- Solution Architect
- Tech Lead
- UX/UI Designer
- Backend Engineer
- Frontend Engineer
- QA Lead / QA Engineer
- DevOps Engineer / SRE
- Release Manager

## 5. Зависимости между этапами
- Discovery блокирует старт Design.
- Design блокирует Development (архитектура и UX/UI).
- Development блокирует QA (доступность фич и стабильных API).
- QA блокирует Release (подтверждение качества).
- Release зависит от готовности DevOps/инфраструктуры.

## 6. Зависимости функциональных сценариев (WBS)
- Пользователи → Проекты (для `create_project` нужен `owner_id`, пользователь должен существовать).
- Проекты → Задачи (для `create_task` нужен `project_id`).
- Задачи → Завершение/возобновление (для `complete_task`/`reopen_task` нужен `task_id`).
- Проекты/задачи → Отчеты (для `create_report`/`run_report` нужны данные по проектам и задачам).
- Файлы/данные → Экспорт/импорт (для `export_data`/`import_data` нужна существующая сущность).
- Уведомления → Онбординг/команды (для `create_notification` нужен `user_id`).

## 7. Матрица зависимостей (Task A → Task B)
- Discovery → Design (уточненные требования и цели).
- Архитектура/UX → Development (без согласованных решений нельзя начать реализацию).
- Development → QA (нужны реализованные фичи и стабильные API).
- QA → Release (только после подтверждения качества).
- Инфраструктура/репозиторий → Backend API/интеграции (готовность среды и CI/CD).
- Backend API/интеграции → Frontend и UX-сценарии (контракт API для UI).
- Пользователи (`create_user`) → Проекты (`create_project`) (нужен `owner_id`).
- Проекты (`create_project`) → Задачи (`create_task`) (нужен `project_id`).
- Задачи (`create_task`) → Завершение задачи (`complete_task`) (нужен `task_id`).
- Данные по проектам/задачам → Отчеты (`create_report`/`run_report`) (нужна база данных для отчета).
- Сущности → Экспорт данных (`export_data`) (нужна существующая сущность для выгрузки).

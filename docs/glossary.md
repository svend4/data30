# Глоссарий терминов

## 1) Краткое введение
Глоссарий фиксирует базовые термины, чтобы у всех участников (бизнес, интеграторы, разработчики)
было единое понимание сущностей, связей и процессов в метаслое. Термины отражают структуру
системы, жизненный цикл функций, сценарии и критерии качества.

## 2) Как пользоваться глоссарием
1. Термины используются как единый словарь для описания функций, сценариев и макросов.
2. Если появляется новый термин, он добавляется в таблицу и связывается с существующими.
3. В описаниях предпочтительно использовать единый стиль: английский идентификатор + русское пояснение.
4. Термины должны быть атомарными, без пересечений по смыслу и дублирования.

## 3) Таблица терминов

| Термин | Определение | Пример |
|---|---|---|
| Function (Функция) | Атомарное действие, выполняемое системой/модулем. | “SendEmail”, “CreateRow”, “ParsePDF”. |
| Module (Модуль) | Источник одной или нескольких функций. | Плагин WordPress, модуль Make.com. |
| Use Case (Сценарий) | Логическая цепочка функций для достижения результата. | “Лид → CRM → Email → Отчет”. |
| Macro (Макрос/Workflow) | Готовый, параметризуемый сценарий автоматизации. | “Форма → CRM → Письмо”. |
| Cluster (Кластер) | Группа сценариев по одной бизнес-вертикали. | “Документооборот”, “Маркетинг”. |
| Trigger (Триггер) | Событие, запускающее сценарий. | “Новая заявка на сайте”. |
| Pipeline | Последовательность шагов в сценарии. | “Ввод → Проверка → Уведомление”. |
| Input Schema | Формат входных данных функции. | JSON с полями email, subject. |
| Output Schema | Формат выходных данных функции. | status, messageId. |
| Error Code | Стандартизированная ошибка функции. | auth_failed, invalid_email. |
| Integration | Связь между модулями/функциями. | CRM ↔ Email сервис. |
| Mapping | Сопоставление полей между системами. | Email → user.email. |
| Connector | Компонент, обеспечивающий интеграцию. | Коннектор Google Sheets. |
| Adapter | Переходник для нестандартных API. | Адаптер для legacy-сервиса. |
| Data Source | Источник данных. | CRM, Google Sheets. |
| Data Sink | Пункт назначения данных. | BI-отчет, база данных. |
| Transformation | Изменение структуры данных. | JSON → CSV. |
| Validation | Проверка корректности входа. | Проверка email. |
| Orchestration | Управление цепочкой функций. | Оркестрация workflow. |
| Dependency | Зависимость модуля от другого. | Плагин требует библиотеку X. |
| Compatibility | Совместимость модулей/версий. | WordPress 6.x. |
| SLA | Гарантии доступности/скорости. | 99.9% uptime. |
| Latency | Задержка выполнения. | 120 ms на запрос. |
| Throughput | Пропускная способность. | 1000 задач/мин. |
| Rate Limit | Лимит запросов. | 100 req/min. |
| Maturity | Степень зрелости компонента. | Prototype → Stable → Recommended. |
| Rating | Оценка качества/полезности. | 4.7/5 в сценарии. |
| Review | Отзыв пользователя. | “Сценарий работает стабильно”. |
| Proven Usage | Подтвержденное использование. | Лог запуска макроса. |
| Reusability | Повторное использование функции. | Функция применяется в 10 сценариях. |
| Discoverability | Легкость поиска функции. | Поиск по цели/категории. |
| Explainability | Пояснение рекомендаций. | “Выбран из-за 95% успешности”. |
| Governance | Политики управления. | Правила модерации. |
| Versioning | История изменений. | Function v1.2. |
| Compatibility Matrix | Таблица совместимостей. | Платформа ↔ версии. |
| Template | Шаблон сценария. | “Onboarding клиента”. |
| Recipe | Конкретный макрос. | “Заявка → CRM → Email”. |
| Library | Набор макросов. | Библиотека для HR. |
| Use Case Map | Карта сценариев. | Диаграмма бизнес-потока. |
| Catalog | Каталог модулей/функций. | Каталог SaaS-инструментов. |
| Registry | Реестр описаний и версий. | Реестр функций. |
| Taxonomy | Иерархия категорий. | Коммуникации → Email. |
| Ontology | Формальная модель связей. | Связи функций и данных. |
| Normalization | Приведение терминов к стандарту. | “SendEmail” вместо “EmailSend”. |
| Compliance | Соответствие требованиям. | GDPR/локальные нормы. |
| Audit Trail | История действий. | Логи выполнения макроса. |

## 4) Правила расширения глоссария
1. Новый термин добавляется только при отсутствии эквивалента в текущем списке.
2. Для каждого термина требуется короткое определение и пример использования.
3. При появлении конкурирующих терминов выбирается один основной, остальные описываются как синонимы.

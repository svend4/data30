# Позиционирование: каталог/платформа функций

## Входные данные брифа (macro: marketing_brief)

### Сегменты
- **Data/ML команды и аналитики**: строят пайплайны и хотят переиспользовать атомарные функции.
- **Инженеры интеграций и платформенные команды**: стандартизируют функции ввода/вывода, коммуникаций и оркестрации.
- **Продакт‑тимы, создающие новые сценарии/продукты**: быстро комбинируют функции для MVP и прототипов.

### Ключевые JTBD
- **«Собрать пайплайн из проверенных функций без переписывания кода»**.
- **«Быстро найти подходящую функцию и понять входы/выходы/ограничения»**.
- **«Собрать MVP‑сценарий из стандартного словаря функций»**.

### Боли
- Непрозрачные интерфейсы функций и отсутствие единого паспорта.
- Дублирование реализации одних и тех же действий в разных командах.
- Сложность согласования терминов и категорий при масштабировании библиотеки.

### Конкуренты
- **Внутренние библиотеки/реестры функций** без единой спецификации.
- **Low‑code/ETL платформы** с закрытыми каталогами компонентов.
- **Пакетные репозитории** (shared libs) без каталогизации и паспорта.

### Альтернативы
- Писать собственные функции и поддерживать их в нескольких командах.
- Использовать готовые конструкторы пайплайнов без прозрачных интерфейсов.
- Документировать функции в разрозненных вики/README.

## Сопоставление с продуктовой сущностью

### Категории функций (FUNCTION_PASSPORT.md, раздел «Категории функций»)
Каталог/платформа функций закрывает все ключевые категории: ввод/вывод, трансформация, хранение, коммуникации, поиск/извлечение, валидация/качество, безопасность, оркестрация. Это формирует единый язык для команд и упрощает навигацию по библиотеке.

### MVP‑словарь функций (FUNCTION_PASSPORT.md, раздел «MVP‑словарь функций»)
В качестве базового набора используются 60 функций (например, read/write, transform, validate, send, store, schedule). Этот словарь задает минимальный стандарт для запуска MVP‑сценариев и расширения каталога без пересборки архитектуры.

## Гипотезы позиционирования

### Гипотеза 1
- **Для кого:** data/ML и аналитические команды.
- **Ключевая выгода:** быстрый доступ к проверенным атомарным функциям с единым паспортом и понятными интерфейсами.
- **Лучше альтернатив:** сокращает время сборки пайплайнов по сравнению с самописными библиотеками и разрозненной документацией.

### Гипотеза 2
- **Для кого:** платформенные и интеграционные команды.
- **Ключевая выгода:** стандартизированный каталог категорий и единая спецификация функций для масштабирования платформы.
- **Лучше альтернатив:** снижает издержки на сопровождение и согласование интерфейсов по сравнению с локальными реестрами.

### Гипотеза 3
- **Для кого:** продуктовые команды, запускающие новые сценарии и MVP.
- **Ключевая выгода:** готовый MVP‑словарь функций для быстрого прототипирования.
- **Лучше альтернатив:** быстрее и дешевле, чем разработка «с нуля» или внедрение тяжелых ETL/low‑code платформ.

## Финальная версия позиционирования

**Каталог/платформа функций** — это единый стандарт и библиотека атомарных функций для data, платформенных и продуктовых команд, которые хотят быстро собирать и масштабировать пайплайны. Платформа дает понятные паспорта функций, категорийную навигацию и MVP‑словарь для сборки сценариев без дублирования кода, что делает ее быстрее и прозрачнее, чем самописные библиотеки или закрытые конструкторы.

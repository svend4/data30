# Метаслой решений: базовая рамка

## 1. Цель системы
**Цель:** метаслой для поиска, интеграции и повторного использования готовых решений.

## 2. Целевая аудитория
* B2B-компании и продуктовые команды.
* Разработчики (backend/frontend/full-stack).
* Интеграторы и системные архитекторы.
* No-code/low-code пользователи (аналитики, операторы, бизнес-пользователи).

## 3. Ключевые сущности
* **Функция** — атомарная единица полезного поведения (например, «проверка адреса»).
* **Модуль** — логически связанный набор функций, упакованный для повторного использования.
* **Сценарий** — последовательность модулей/функций, описывающая бизнес-процесс.
* **Макрос** — шаблонная последовательность действий, параметризуемая под конкретный кейс.
* **Кластер** — группа решений/модулей, объединённых по домену, задаче или отрасли.
* **Рейтинг** — агрегированная оценка качества и пригодности решения.

## 4. Глоссарий терминов

| Термин | Определение | Пример |
|---|---|---|
| **Метаслой** | Надстроечный уровень, который описывает, связывает и индексирует решения в каталоге. | Метаслой связывает `Function` «проверка адреса» с `UseCase` «онбординг клиента». |
| **Решение** | Оформленный набор артефактов, закрывающий конкретную задачу. | Решение «KYC-проверка» включает функции валидации и модуль отчётности. |
| **Артефакт** | Файл, контракт, скрипт или компонент, входящий в состав решения. | Артефакт: OpenAPI-спецификация сервиса проверки адресов. |
| **Function (Функция)** | Минимальная единица логики, выполняющая одну операцию. | Function «normalize_address» приводит адрес к стандартному формату. |
| **Module (Модуль)** | Набор функций, объединённых единым контекстом и повторно используемых вместе. | Module «address_validation» включает функции нормализации и геокодирования. |
| **UseCase (Сценарий)** | Бизнес-сценарий, описывающий цепочку модулей/функций для результата. | UseCase «онбординг клиента» вызывает модули KYC и валидации адреса. |
| **Macro (Макрос)** | Повторяемый шаблон сценария с параметрами, который можно быстро настроить. | Macro «массовая проверка адресов» параметризуется источником данных. |
| **Cluster (Кластер)** | Доменная группа связанных решений или модулей. | Cluster «Финтех» объединяет решения KYC, скоринга и antifraud. |
| **Каталог** | Структурированное хранилище решений и метаданных. | В каталоге хранится решение «KYC-проверка» с версиями и рейтингом. |
| **Индекс** | Набор поисковых ключей и связей между сущностями. | Индекс связывает UseCase «онбординг» с функциями валидации. |
| **Метаданные** | Описательные атрибуты решения: версия, автор, лицензия, теги. | Метаданные решения включают версию 1.2.0 и лицензию MIT. |
| **Тег** | Короткая метка для классификации сущностей. | Тег «antifraud» добавлен к модулю скоринга. |
| **Таксономия** | Иерархия категорий и связей для навигации по каталогу. | В таксономии «Финтех → Риски → Скоринг» найден модуль. |
| **Онтология** | Формальная модель сущностей и их отношений в системе. | Онтология описывает связь `UseCase → Macro → Function`. |
| **Контракт** | Интерфейс взаимодействия: API, событие, формат данных. | Контракт API `/validate` определяет входные поля адреса. |
| **Integration (Интеграция)** | Соединение решения с внешними системами и источниками данных. | Интеграция с CRM передаёт данные клиента в модуль KYC. |
| **Коннектор** | Технический компонент для интеграции с источником/системой. | Коннектор к SAP выгружает данные клиентов в пайплайн. |
| **Совместимость** | Степень согласованности решений по форматам и протоколам. | Модуль отчётности совместим с JSON-схемой v2. |
| **Повторное использование** | Применение решения в новом контексте без переработки логики. | Функция геокодирования повторно используется в модуле доставки. |
| **Композиция** | Сборка нового решения из существующих модулей и функций. | Композиция KYC и скоринга образует решение «кредитный лимит». |
| **Оркестрация** | Управление исполнением сценариев, зависимостями и порядком шагов. | Оркестратор запускает модуль KYC перед модулем скоринга. |
| **Шаблон** | Типовой каркас решения, ускоряющий сборку под новый кейс. | Шаблон «интеграция с CRM» задаёт структуру коннекторов. |
| **Pipeline (Пайплайн)** | Последовательность шагов обработки данных или процесса. | Pipeline «ETL клиентов» включает загрузку, нормализацию и проверку. |
| **Trigger (Триггер)** | Условие или событие, запускающее сценарий или шаг пайплайна. | Trigger «новый клиент в CRM» запускает UseCase онбординга. |
| **Rating (Рейтинг)** | Числовая агрегированная оценка пригодности и качества решения. | Rating 4.7 повышает видимость решения в каталоге. |
| **Maturity (Зрелость)** | Уровень зрелости решения по стабильности, тестированию и эксплуатации. | Maturity «Production» означает покрытие тестами и SLA. |
| **Отзыв** | Качественная оценка пользователей о применимости решения. | Отзыв отмечает, что модуль работает стабильно в пиковые часы. |
| **Профиль решения** | Карточка с параметрами, зависимостями, метриками и рейтингом. | В профиле решения указаны версии модулей и SLA. |
| **Версионирование** | Управление изменениями и совместимостью решений во времени. | Версионирование фиксирует переход модуля на 2.0.0. |
| **Лицензия** | Правовые условия использования решения. | Лицензия Apache-2.0 разрешает коммерческое использование. |
| **Безопасность** | Требования и меры защиты данных и доступа. | Политика безопасности требует шифрования PII. |
| **Соблюдение требований** | Соответствие регуляторике и внутренним стандартам. | Модуль KYC соответствует требованиям AML. |
| **Мониторинг** | Контроль состояния и метрик использования решения. | Мониторинг отслеживает время ответа функций KYC. |
| **Телеметрия** | Сбор данных о работе решения (логи, метрики, трассировки). | Телеметрия пишет latency каждого шага пайплайна. |
| **SLA** | Соглашение об уровнях сервиса и целевых показателях. | SLA 99.9% доступности для модуля проверки адресов. |
| **Наблюдаемость** | Прозрачность поведения системы через логи, метрики и трейсы. | Наблюдаемость позволяет найти узкое место в макросе. |
| **Стоимость владения** | Совокупные затраты на внедрение и поддержку решения. | Стоимость владения снижена за счёт повторного использования. |

## 5. Рамка ценности
**Переход от хаоса к управляемости:**
* **Хаос:** разрозненные решения, дублирование, неясное качество, трудно интегрировать.
* **Управляемость:** единый каталог, проверенные модули, прозрачные зависимости, быстрый поиск и повторное использование.
* **Критичность:** сокращение времени внедрения, снижение рисков интеграции, повышение качества за счёт повторяемости.
